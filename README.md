二、产品概述
2.1 产品定位
基于微信社交生态的同城年轻人轻户外拼团工具，核心解决“缺乏玩伴、组织麻烦、决策低效、传统跟团体验差”的痛点，为18-35岁青年提供“找对人、走野的路，一键开团，说走就走”的周边轻旅行服务，定位为“社交+拼团+大巴调度”的中介型工具，而非传统旅行社或深度旅游平台。
2.2 Slogan
找对的人，走野的路，一键开团，说走就走。
2.3 核心价值
- 用户价值：快速匹配契合的旅伴，无需自行组织、定路线、找车辆，降低周边游门槛，提升旅行社交体验，规避传统跟团的刻板与麻烦。
- 商业价值：通过大巴运力差价实现核心盈利，后期延伸供应链（门票、保险等）赚取佣金，依托微信生态实现低成本获客与裂变。
- 运营价值：以轻量MVP快速跑通全流程，验证需求真实性，积累种子用户与合作车队资源，为后续迭代奠定基础。
2.4 核心业务流程（用户旅程）
注册打标签 → AI匹配/自建群 → 群内“三选一”路线投票 → 达成门槛/一键锁车 → AA支付车费 → 线下定点上车 → 游后评价沉淀
三、核心功能需求（MVP版本）
MVP版本聚焦“核心流程闭环”，不做冗余功能，优先实现“匹配-建群-决策-支付-履约-评价”六大核心环节，具体分为4个层级：社交与匹配层、群组与决策层、履约与交易层、行中与行后体验层。
3.1 社交与匹配层（引流端）
3.1.1 注册与旅行人格测试
功能描述
用户首次进入小程序，需完成微信授权注册（绑定微信昵称、头像），随后强制参与趣味旅行人格测试，通过问答生成个人旅游DNA标签，作为后续AI匹配与路线推荐的核心依据。
核心需求
快速完成用户画像标签化，为精准匹配旅伴、生成适配路线提供数据支撑，同时提升注册趣味性，降低用户流失。
详细需求
- 注册流程：微信授权 → 绑定手机号（可选，用于找回密码、接收行前提醒） → 进入人格测试。
- 人格测试：包含5-8道趣味选择题，无标准答案，示例如下：
        
  - 旅行节奏：早鸟型（早起赶行程）vs 睡到自然醒（不赶时间）
  - 旅行强度：特种兵（一天打卡多个景点）vs 躺平派（慢逛放松）
  - 旅行偏好：出片狂魔（优先拍照打卡）vs 深度体验（感受当地氛围）
  - 饮食偏好：当地特色小吃 vs 常规家常菜
  - 同行偏好：安静独处 vs 热闹互动
- 标签生成：测试结束后，自动生成3-5个核心标签（如“睡到自然醒+深度体验+特色小吃”），标签不可手动修改，可在个人中心查看。
- 跳过限制：测试不可跳过，若中途退出，再次进入小程序需重新开始测试。
交互逻辑
微信授权后自动跳转至测试页面，每道题单选，点击“下一题”进入下一题，完成最后一题后，1秒内生成标签，跳转至首页。
3.1.2 AI旅伴推荐
功能描述
基于用户的LBS定位（同城优先）+ 旅游DNA标签匹配度，系统自动推荐契合度≥80%的其他用户，支持右滑加好友、左滑跳过，添加好友后可直接发起建群或加入对方创建的群组。
核心需求
解决“缺乏玩伴”的核心痛点，通过精准匹配，让用户快速找到兴趣契合的旅伴，提升社交转化率。
详细需求
- 定位权限：首次使用需申请微信地理位置权限，定位精度为“城市+区域”（如“成都-锦江区”），无需精确到具体街道，保护用户隐私。
- 匹配规则：优先推荐同城用户，标签匹配度≥80%，匹配度越高，排序越靠前；若同城匹配用户不足（少于10人），可扩大至周边城市。
- 用户展示：推荐卡片展示用户昵称、头像、核心旅游标签、匹配度百分比，不展示其他隐私信息（如手机号、具体位置）。
- 交互操作：右滑=添加好友，左滑=跳过该用户；添加好友后，弹出“添加成功，可发起建群/加入对方群组”的提示，跳转至“建群”页面或“群组列表”页面。
- 限制：每天最多推荐20个用户，避免用户选择疲劳；已添加的好友不再重复推荐。
3.1.3 发现页（瀑布流）
功能描述
首页默认展示发现页，采用小红书式瀑布流布局，展示当前“正在招募中的群组”，吸引用户加入，同时作为群组裂变的核心入口。
核心需求
提升群组曝光率，降低用户建群后的招募难度，同时为被动寻找玩伴的用户提供入口，提升平台活跃度。
详细需求
- 内容展示：每个群组卡片包含“群组名称、发起者头像+昵称、核心路线类型（如“周边爬山+农家乐”）、当前人数/目标人数、拼团倒计时、集合城市”。
- 排序规则：优先展示“即将截止倒计时”（剩余时间越少，排序越靠前），其次展示“人数接近目标值”的群组，最后按群组创建时间排序。
- 交互操作：点击群组卡片，进入群组详情页，可查看群成员标签、投票进度、拼团状态，点击“加入群组”即可加入（无需群主审核，MVP版本简化流程）。
- 筛选功能：支持按“路线类型”（爬山、古镇、露营等）、“距离”（同城、周边）筛选群组，筛选按钮固定在发现页顶部。
- 空状态：若当前无正在招募的群组，展示“暂无招募群组，快来发起你的第一群吧”的提示，附带“发起建群”按钮，跳转至建群页面。
3.2 群组与决策层（转化端 - 核心亮点）
3.2.1 自建群/加入群
功能描述
用户可自主发起建群，设置群组基本信息；也可通过发现页、AI匹配好友推荐，加入其他正在招募的群组，群组支持百人超大群，方便用户拉人扩列。
核心需求
提供用户聚集的场景，为后续路线投票、拼团履约奠定基础，同时支持用户自发裂变，降低获客成本。
详细需求
- 自建群流程：点击“发起建群” → 填写群组名称（如“周末成都周边爬山群”）、选择预计出行时间（周末/单日）、设置目标人数（默认50人，可调整为30-100人） → 生成群组，自动成为群主 → 可分享群组至微信好友/朋友圈，邀请他人加入。
- 群组权限：群主可踢人、修改群组基本信息、发起路线投票、查看拼团进度；普通群成员可投票、邀请好友加入、支付参团费用、查看群内公告。
- 百人超大群：支持最多100人加入，超过100人无法继续加入，群主可提前调整目标人数；群成员可自由邀请微信好友加入，无需群主审核（MVP版本简化流程）。
- 群组列表：个人中心设置“我的群组”入口，展示“我创建的群”“我加入的群”，区分“招募中”“投票中”“拼团成功”“拼团失败”四种状态，方便用户快速查看。
3.2.2 群内路线投票卡片（核心设计）
功能描述
群主发起投票后，系统根据群成员的旅游DNA标签大数据，自动生成3条半标准化路线，每条路线附带详细信息，群成员在规定时间内投票，票数最高的路线作为最终出行路线，禁止用户自行填写路线。
核心需求
解决群内“决策瘫痪”的痛点，通过系统生成标准化路线，减少用户沟通成本，同时保证路线适配群成员偏好，提升用户体验。
详细需求
- 投票发起：群主可在群组内点击“发起投票”，系统自动触发路线生成，无需群主手动添加路线；投票发起后，群内弹出投票卡片，所有群成员可见。
- 路线生成规则：基于群内所有成员的核心标签，生成3条差异化路线（如群成员多为“出片狂魔+躺平派”，则生成“古镇打卡+咖啡馆”“海景露营+烧烤”“文创街区+下午茶”3条路线），路线类型以周边单日游为主（车程≤2小时）。
- 路线信息：每条路线需包含“路线名称、预估总时长（如8:00-18:00）、集合地（2-3个核心地铁口/公交枢纽，系统根据群成员常住地聚类生成）、人均大巴费预估（精确到元，包含平台抽成）、路线简要描述（1-2句话，如“全程轻松无徒步，适合拍照打卡，含农家乐午餐”）。
- 投票规则：每个群成员仅可投1票，投票不可修改；投票时长默认24小时，群主可调整为12-48小时；未投票的成员，在投票结束时自动视为“跟随多数派”，计入票数最高的路线。
- 投票结果：投票结束后，系统自动弹出结果提示，明确最终路线，同时更新群组状态为“投票结束，待拼团”，并展示最终路线详情。
- 限制：每个群组仅可发起1次路线投票，投票结束后不可重新发起；若投票人数不足群成员的50%，则视为投票无效，需重新发起投票。
3.3 履约与交易层（闭环端）
3.3.1 拼团倒计时与门槛
功能描述
路线投票结束后，自动进入拼团阶段，设置拼团门槛（如“满30人即发车”）和48小时倒计时，若倒计时结束前达成人数门槛，则拼团成功；若未达成，则自动退款、解散群组，避免用户尴尬。
核心需求
保障大巴运力的利用率，降低平台运营成本，同时明确拼团规则，减少用户纠纷。
详细需求
- 拼团门槛：默认设置“满30人发车”，群主可根据目标人数调整（最低30人，最高50人，与大巴座位数匹配）；拼团门槛需在投票结束后明确告知所有群成员。
- 倒计时：拼团倒计时默认48小时，从投票结束后立即开始，倒计时实时展示在群组顶部，精确到分秒；倒计时结束前2小时，系统自动发送提醒（群内通知+微信服务通知）。
- 拼团状态：实时展示“当前人数/目标人数”，如“25/30”，同时标注“距离拼团结束还有XX小时XX分”；达成门槛后，自动切换为“拼团成功，待支付”状态；未达成则切换为“拼团失败，自动退款”状态。
- 拼团失败处理：倒计时结束未达成人数门槛，系统自动解散群组，同时向所有已支付的用户发起全额退款（退款到原支付账户，到账时间遵循微信支付规则，一般1-3个工作日），并发送退款通知。
3.3.2 大巴运力匹配（人工指派）
功能描述
MVP阶段不做复杂API对接，后端运营人员维护本地合规大巴车队库，群组成团成功后，运营人员在后台人工指派车辆，将司机信息、车牌号、车辆照片推送到小程序群组内，同步通知所有群成员。
核心需求
简化MVP阶段的技术复杂度，快速实现履约环节，同时保障车辆合规性，降低安全风险。
详细需求
- 后台车队库：运营后台维护3-5家合规大巴车队信息，包含车队名称、营运资质、保险信息、车辆型号（50座大巴）、司机信息（姓名、联系方式、从业资格证）、车辆照片。
- 人工指派：拼团成功后，运营人员在后台根据出行路线、集合地，选择合适的车队和车辆，完成指派后，系统自动将“车牌号、司机姓名、司机电话、车辆照片”推送到群组内，形成“车辆信息卡片”。
- 信息通知：车辆信息推送后，向所有群成员发送微信服务通知，提醒“拼团成功，车辆信息已公布，请查看”；群内车辆信息卡片可随时查看，不可修改。
- 异常处理：若指派的车辆出现故障，运营人员需在1小时内重新指派车辆，并更新群组内的车辆信息，同时向所有群成员发送通知，说明情况。
3.3.3 AA支付与锁定
功能描述
拼团成功后，群成员点击“确认参团”，系统自动计算人均大巴费，支持微信AA支付，支付金额由平台暂存，行程结束后结算给大巴公司；未支付的成员，视为放弃参团，自动移出群组。
核心需求
解决“收钱难、信任差”的问题，保障平台、用户、大巴公司的三方权益，实现交易闭环。
详细需求
- 费用计算：人均大巴费=（大巴底价+平台抽成）÷ 拼团人数，精确到元，平台抽成比例为10%-15%（MVP版本固定为12%，后台可调整）；费用明细在支付页面明确展示（大巴底价、平台抽成、人均费用）。
- 支付流程：拼团成功后，群内弹出“确认参团并支付”按钮，用户点击后，跳转至微信支付页面，完成支付；支付成功后，系统提示“支付成功，已锁定参团名额”，同时更新群成员状态为“已支付”。
- 支付时限：拼团成功后，用户需在24小时内完成支付，逾期未支付的，系统自动将其移出群组，释放参团名额，同时发送“逾期未支付，已取消参团资格”的通知。
- 平台暂存：用户支付的费用由平台暂存，行程结束后（用户确认上车并完成行程），平台在24小时内将扣除抽成后的费用结算给大巴公司；若行程取消（如车辆故障、天气原因），平台全额退还用户支付的费用。
- 退款规则：支付后，若用户因个人原因无法参团，需在行程前24小时申请退款，可全额退款；行程前24小时内申请退款，扣除20%手续费；行程当天不支持退款；退款申请在个人中心提交，运营人员手动审核，审核通过后1-3个工作日到账。
3.4 行中与行后体验层
3.4.1 行前提醒
功能描述
行程出发前1天，系统自动向所有已支付的群成员发送行前提醒卡片，包含集合时间、集合地、天气情况、必备物品清单，同时通过微信服务通知同步提醒。
核心需求
提升用户出行体验，减少因信息遗漏导致的延误、遗漏物品等问题，降低用户投诉率。
详细需求
- 提醒时间：行程出发前1天18:00，自动发送行前提醒卡片至群组内，同时推送微信服务通知。
- 提醒内容：明确集合时间（如“明日8:00集合，8:30准时发车，过时不候”）、集合地（2-3个核心接驳点，标注具体地铁口/公交枢纽名称）、当日天气（温度、晴雨情况）、必备物品（如“舒适运动鞋、防晒用品、充电宝、少量现金”）、注意事项（如“自行前往集合地，不可迟到”）。
- 交互：行前提醒卡片可收藏、转发，群成员可在卡片下方留言提问，群主或运营人员可回复。
3.4.2 虚拟大巴票
功能描述
行程出发前，系统为每个已支付的用户生成带唯一二维码的虚拟大巴票，包含用户信息、行程信息、车辆信息，上车时可通过扫码核销（或司机查看名单放行），作为上车凭证。
核心需求
简化上车核验流程，保障参团用户的权益，防止非参团人员上车，同时便于司机核对人数。
详细需求
- 车票生成：行程出发前6小时，系统自动为已支付用户生成虚拟大巴票，可在个人中心“我的订单”中查看。
- 车票信息：包含用户昵称、头像、旅游标签、行程名称、出发时间、集合地、车牌号、司机信息、唯一核销二维码。
- 核销方式：MVP版本支持两种核销方式，可后台切换：
        
  - 扫码核销：司机通过平台后台生成的核销码，扫描用户虚拟车票的二维码，核销成功后，系统提示“核销成功”。
  - 名单核对：司机后台可查看参团用户名单（包含昵称、头像），核对用户身份后，手动标记核销。
- 异常处理：若用户虚拟车票丢失或二维码无法识别，可在个人中心重新生成，或联系司机核对名单。
3.4.3 群相册与游后评分
功能描述
行程中，群成员可上传旅行照片至群相册，共享旅行瞬间；行程结束后，系统自动发起游后评分，用户可对旅伴、行程路线、大巴服务进行打分，评分结果沉淀为用户标签和平台参考。
核心需求
提升用户社交体验，同时通过评分机制筛选优质旅伴、优质车队，规避“放鸽子、奇葩旅伴”等问题，优化平台服务质量。
详细需求
- 群相册：行程开始后，群组内自动生成群相册，群成员可上传照片（最多9张/次）、添加文字描述，可点赞、评论；行程结束后，群相册保留7天，7天后自动归档至个人中心“我的旅行”。
- 游后评分：行程结束后24小时内，系统自动向所有参团用户发送评分提醒（群内通知+微信服务通知），用户需完成评分后，方可查看他人评分。
- 评分内容：分为3个维度，每个维度1-5星评分，可添加文字评价（可选）：
  - 旅伴匹配度：评价同行旅伴是否与标签契合、是否有不文明行为（如放鸽子、吵闹等）。
  - 行程路线：评价路线是否符合预期、集合地是否便捷、行程节奏是否合理。
  - 大巴服务：评价车辆车况、司机服务态度、行车安全。
- 评分沉淀：评分结果将影响用户标签（如多次被评为“不文明旅伴”，将添加“失信”标签，降低匹配优先级）和车队评分（低于4分的车队，后台限制派单）；评分内容可在群组详情页、个人中心查看（匿名展示，不泄露评分人信息）。
四、非功能需求
4.1 性能需求
- 加载速度：小程序启动时间≤3秒，页面切换时间≤1秒，图片加载时间≤2秒（瀑布流图片可懒加载）。
- 并发能力：支持同时在线1000人，群组内消息发送、投票、支付等操作无延迟，无卡顿。
- 稳定性：小程序运行过程中无崩溃、无闪退，支付流程成功率≥99%，核销流程成功率≥99%。
4.2 兼容性需求
支持微信iOS版本8.0及以上、Android版本8.0及以上，适配不同尺寸的手机屏幕（手机、平板），无排版错乱、功能异常问题。
4.3 安全需求
- 用户隐私：严格保护用户个人信息（手机号、地理位置、身份证信息等），不泄露、不滥用；仅在用户授权后，方可获取相关信息。
- 支付安全：对接微信支付官方接口，保障支付流程安全，防止支付漏洞、资金被盗等问题；平台暂存资金需单独管理，确保资金安全。
- 数据安全：用户数据、群组数据、交易数据需定期备份，防止数据丢失；后台操作需记录日志，可追溯操作记录。
4.4 易用性需求
- 操作简单：核心流程（注册、匹配、建群、投票、支付）操作步骤≤3步，符合微信小程序用户使用习惯，无需额外学习。
- 反馈清晰：所有操作（支付、投票、核销等）均有明确的成功/失败提示，错误操作有引导性提示（如“支付失败，请检查网络后重试”）。
- 导航清晰：小程序底部导航栏固定（首页、发现、我的群组、个人中心），用户可快速切换页面，找到所需功能。
五、风控与避坑细则（生死线）
5.1 安全风控（一票否决项）
- 实名制要求：所有参团用户必须完成实名制验证，MVP版本接入微信实名接口，若微信未实名，需要求用户上传身份证OCR（仅用于购买保险和身份核验，不存储完整身份证信息）；未完成实名制的用户，不可支付参团，不可上车。
- 强制保险：每一笔大巴订单，必须强制捆绑一份一日游意外险（单价≤5元），费用包含在人均大巴费中，无需用户额外支付；保险信息需在支付页面明确展示，用户可查看保险条款。
- 防灰黑产：后台设置风控规则，严防大巴车队脱离平台私下接单（飞单），若发现车队飞单，立即终止合作，并扣除保证金；禁止用户恶意刷单（如重复建群、虚假支付），一经发现，封禁账号，限制使用平台功能。
5.2 接送点优化（降低成本+筛选用户）
- 禁止用户自行填写接送地址，系统根据群成员常住地（通过微信定位聚类），自动生成2-3个最合理的“地铁始发站/大型公交枢纽”作为接驳点，在路线信息中明确告知。
- 规则明确：在行程提醒、路线详情中，明确标注“请自行前往接驳点，过时不候”，筛选掉无时间观念的用户，同时降低大巴接送成本。
5.3 群主权责界定
- 平台规则明确：在用户注册、建群时，弹出用户协议，明确“群主仅为旅行发起人，不承担旅行社相关责任，所有合同关系、车辆安全责任，由平台+大巴公司直接与参团成员签订”。
- 免责提示：群组内、行程详情页，显著位置展示免责提示，明确群主无安全保障、行程安排等相关责任，避免后续纠纷。
5.4 决策瘫痪处理
- 投票倒计时强制：投票时长最长不超过48小时，最短不低于12小时，未投票用户自动视为“跟随多数派”，避免投票无限期拖延。
- 投票提醒：投票期间，系统每8小时发送一次投票提醒（群内通知），提升投票率，避免因用户遗忘导致投票无效。
5.5 车队管理（保障服务质量）
- 人工审核：MVP阶段，运营人员对合作大巴车队进行重度审核，核查营运证、车辆保险、司机从业资格证，确保车队合规；未通过审核的车队，禁止接入平台。
- 评分机制：行程结束后，用户对大巴服务的评分将作为车队考核依据，评分低于4分的车队，系统限制派单；连续3次评分低于4分，终止合作。
六、运营与商业模式补充（MVP版本）
6.1 盈利模式
- 核心盈利：大巴运力差价/抽成，平台从大巴公司拿到底价，卖给用户市场价，抽成比例为10%-15%（MVP版本固定为12%，后台可调整）。
- 后期延伸：MVP版本暂不接入，后续迭代可接入门票、午餐、旅游意外险等供应链资源，赚取佣金。
6.2 冷启动策略
- 区域聚焦：MVP版本仅开放一个城市（成都/杭州/广州，选择周边游发达城市），集中资源跑通流程，不做全量推广。
- 供给端：提前谈下3-5家靠谱的合规大巴车队，承诺为其导流，签订合作协议，明确底价、抽成比例、服务标准。
- 需求端：前往豆瓣同城群、小红书、大学城论坛，寻找现有“户外俱乐部群主”“飞盘群主”，邀请其入驻平台建群，给予免单、佣金奖励（如成功发起拼团并发车，给予群主50元佣金）。
七、MVP版本里程碑
1. 第1-2周：完成小程序开发（核心功能全部实现）、后台管理系统开发（车队管理、订单管理、核销管理、用户管理）、合作车队对接。
2. 第3周：完成内部测试（功能测试、兼容性测试、安全测试），修复bug；邀请10-20名种子用户进行内测，优化操作体验。
3. 第4周：正式上线MVP版本，启动冷启动策略，招募群主建群；周末组织50个真实用户，补贴大巴费，跑通“线上建群-投票-支付-线下发车-游后评分”全流程。
4. 里程碑验收标准：50名真实用户完成线下出行，用户满意度≥80%，全流程无重大bug、无安全事故、无用户投诉。
八、用户协议要点（核心条款）
- 用户需完成实名制验证，方可参团；不得提供虚假信息，否则平台有权封禁账号。
- 参团用户需遵守平台规则，不得在群组内发布违规内容、恶意骚扰他人，否则群主可踢人，平台有权封禁账号。
- 明确群主仅为发起人，不承担旅行相关责任，平台与大巴公司承担相应的安全、服务责任。
- 用户支付参团费用后，需遵守退款规则，逾期未退款或违规退款，平台有权拒绝退款。
- 平台有权根据运营情况，调整拼团规则、抽成比例、路线类型等，调整后将通过小程序通知用户。
九、后续迭代规划（MVP之后）
MVP版本跑通全流程后，后续迭代重点围绕“提升体验、拓展功能、扩大规模”展开，核心迭代方向如下：
- 1.0→1.5版本：优化AI匹配算法，提升旅伴匹配精准度；接入更多大巴车队，实现运力自动匹配（替代人工指派）；新增路线评价沉淀，优化路线推荐逻辑。
- 1.5→2.0版本：拓展供应链（门票、午餐、保险），增加盈利点；开放多城市运营，实现跨城市周边游；新增用户等级体系、积分兑换功能，提升用户粘性。
- 2.0之后：接入短视频功能，提升社交体验；推出“私人定制团”功能，满足小众需求；优化风控系统，实现全流程自动化风控。
